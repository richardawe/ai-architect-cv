Legal-Tech · Agentic Phase 1 Live

EU261 Auto-Claim Agent

Lead AI Architect & Developer

Agentic system that autonomously files EU Regulation 261/2004 flight delay and cancellation compensation claims — intake, eligibility evaluation, demand letter generation, airline submission, and National Enforcement Body escalation, all orchestrated via GitHub Actions.

GitHub ↗
  • GitHub Actions as distributed state machine — 11 discrete event-driven workflows (intake, eligibility, draft, submit, escalate…) each firing on GitHub Issue label transitions: claim → state:intake-complete → state:eligibility-passed → state:draft-ready → submitted. No external queue or database needed.
  • Hybrid AI/rules architecture — engine/eligibility.py is a deterministic Python rules engine reading versioned YAML thresholds from rules/. The LLM (engine/extractor.py) is used only for structured fact extraction from free-text narratives — never for legal decisions.
  • libsodium PII encryption — engine/crypto.py rewrites PII out of the public Issue body during intake; NACL_PUBLIC_KEY stored as a GitHub repo variable.
  • PAT workaround pattern — intake.yml uses CLAIM_CREATION_TOKEN (a PAT) rather than GITHUB_TOKEN when adding state:intake-complete, because GitHub silences label events triggered by the default token, which would break the downstream eligibility workflow trigger.
  • Pluggable adapter architecture — adapters/ holds per-airline YAML configs; adapters_runtime/ implements Playwright (web-form), SMTP, and HTTP runtimes, decoupled from the engine.
  • JWT consent tokens gate the JSON API (api/) for agent-to-agent callers; OpenAPI spec documented in docs/agent-api.md. Full pytest suite covers engine and workflows_lib.
Python 3.12 · GitHub Actions (11 workflows) · YAML rules engine · libsodium · JWT · Playwright · pytest
Political Intelligence · Cron Pipeline Live — 6h updates

UK Politics AI Positioning Matrix

Lead AI Architect & Developer

Live political intelligence dashboard monitoring UK news and mapping party ideological positions onto a dynamic 2×2 political compass (left/right × authoritarian/libertarian). Tracks 6 major parties with weekly movement indicators, trend charts, and AI commentary. Deployed as a static Next.js site on GitHub Pages updated autonomously every 6 hours.

GitHub ↗
  • Scheduled cron pipeline — GitHub Actions cron every 6 hours runs: fetchNews → extractSignals → calculatePositions → generateCommentary → saveSnapshots → commit → deploy Pages. Concurrency group pages prevents race-condition half-deploys.
  • Multi-provider AI abstraction — lib/ai/index.ts getAIProvider() resolves to OpenRouter, OpenAI, or Anthropic at runtime from env vars. All three adapters share the same prompts, making the LLM entirely swappable with no code changes.
  • Credibility-weighted signal scoring — SOURCE_WEIGHTS rank sources from official_speech=1.0 and hansard=0.95 down to social_media=0.2. SIGNAL_TYPE_MULTIPLIERS weight legislation=1.0 vs rhetoric=0.5 vs signalling=0.4. POLICY_AXIS_WEIGHTS split each policy area into social vs economic impact.
  • Baseline anchoring & reversion — party positions are anchored to 2024 general election manifesto baselines; REVERSION_RATE=0.05 per update cycle prevents drift from ephemeral news. A single-day movement cap blocks implausible position jumps.
  • Cold-start seeding — generateHistory.ts bootstraps 30 days of synthetic history on first deploy, giving the timeline chart meaningful data without live API calls.
  • Fully static output — Next.js App Router exported to GitHub Pages; data served as committed JSON snapshots; no server, no database, zero hosting cost.
Next.js (App Router) · TypeScript · React · Recharts · Tailwind CSS · OpenRouter/OpenAI/Anthropic adapters · GitHub Pages

EV Showcase — Electric Vehicles of the World

Lead Developer

Comprehensive, searchable showcase of every electric vehicle globally — cars, trucks, motorcycles, buses — with specs, range, battery, and pricing. Database grown autonomously: a daily GitHub Actions cron adds newly confirmed EV models via LLM and fills vehicle photos from Wikipedia and Wikimedia Commons.

GitHub ↗
  • Two-job GitHub Actions workflow — Job 1 (update-data) runs the Python pipeline and commits vehicles.json only if changes exist via changed=true/false output gating, preventing spurious commits. Job 2 (deploy) always runs after and deploys via OIDC.
  • LLM as structured data source — update_vehicles.py calls openai/gpt-oss-120b:free via OpenRouter with the full existing vehicle list as context; the model identifies up to 10 new confirmed EVs per run and returns typed JSON.
  • Chain-of-thought stripping — extract_json() removes <think>/<reasoning>/<thought> tags emitted by reasoning models before JSON parsing, enabling compatibility with DeepSeek R1 and similar.
  • Schema validation before commit — every LLM-returned vehicle is checked for required fields, valid type enum, positive numeric specs, and year in 2010–2030. Invalid entries are logged and rejected without halting the pipeline.
  • Three-tier Wikipedia image pipeline — ARTICLE_MAP maps vehicle IDs to exact Wikipedia article titles; fallback 1: Wikipedia article search; fallback 2: Wikimedia Commons file-namespace search with make/model scoring. SVGs, logos, and non-free files are rejected.
  • Jamstack architecture — pure static HTML/CSS/JS reads vehicles.json at load time; no runtime server, no browser API calls, near-zero hosting cost.
Python 3.12 · OpenRouter (gpt-oss-120b:free) · Wikipedia/Wikimedia APIs · GitHub Actions (daily cron) · Static HTML/CSS/JS · GitHub Pages
Browser-Native · Video Generation Live

Brand Reel — Webpage to Brand Video

Lead Developer

Browser-native tool that takes any webpage (HTML upload or screenshot) and generates a polished 10-second brand video entirely client-side — no server, no upload, no API key. Automatically detects page sections, extracts brand accent colour from computed CSS, captures scroll-position screenshots, and renders an animated video.

Live Site ↗ GitHub ↗
  • Scene composition engine — BrandVideo.tsx uses Remotion AbsoluteFill + Sequence primitives for a typed, frame-accurate composition: Intro (90 frames/3s) → Feature scenes (150 frames/5s each, up to 5) → Outro (90 frames/3s). Timing constants in types.ts are shared between the Remotion preview and the canvas exporter.
  • Client-side page analysis — HTML file loaded into a hidden 1280×720 iframe; 13 semantic CSS class selectors ([class*=hero], [class*=feature], [class*=pricing], etc.) detect sections with 100px-bucket deduplication and Y-interval fallback when fewer than 2 sections are found.
  • Accent colour extraction — inspects getComputedStyle on buttons/CTAs/links, rejects near-white (R/G/B >235) and near-black (<20), returns the first valid hex. Falls back to #6366f1.
  • Canvas 2D frame renderer — custom remap() + easing functions (easeOut, easeSine, easeInOut) replicate CSS animation easing in canvas math: opacity fades, Ken Burns zoom, text slide-in, animated accent bars, progress dot indicators.
  • MediaRecorder video encoding — canvas.captureStream(30fps) feeds MediaRecorder at 8 Mbps; codec preference is VP9 (video/webm;codecs=vp9), falling back to generic WebM. Frames committed at real-time pace to keep the stream clock accurate.
  • Dual render paths — Remotion for in-browser preview with scrubbing; Canvas 2D + MediaRecorder for the downloadable .webm, decoupling preview quality from export fidelity.
  • GitHub Actions deploy — Vite build in remotion-brand-video/, OIDC deploy to Pages (id-token: write) — no secrets required.
React · TypeScript · Vite · Remotion · html2canvas · Canvas 2D API · MediaRecorder API (WebM/VP9) · Tailwind CSS · GitHub Pages
Browser-Native ML · Zero Server Live

BrowserML — Client-Side AI Task Runner

Lead Developer

Browser-native AI task runner for text summarisation, creative writing, extractive Q&A, and multi-language translation — running 100% client-side. No server, no API key, no data sent anywhere. Models downloaded once and cached in the browser.

GitHub ↗
  • Web Worker isolation — all Transformers.js inference runs in a dedicated worker thread via postMessage/addEventListener, keeping the main UI thread fully responsive. Progress events are streamed back to drive a live progress bar.
  • Transformers.js model roster — distilbart-cnn-6-6 (summarisation), gpt2 (text generation), distilbert-base-uncased-distilled-squad (extractive QA); translation uses opus-mt-{lang-pair} loaded dynamically by language pair.
  • Browser model cache — env.useBrowserCache=true stores model weights (85–300 MB) in the browser's Cache API after first download for instant reuse on repeat visits.
  • Pipe reuse pattern — getPipe() caches initialised pipeline instances in a pipes{} object keyed by task/model name, avoiding redundant reinitialisation within a session.
  • Request ID correlation — each worker message carries an id generated by the caller, allowing the UI to match async responses to the correct pending request and discard stale results.
  • Zero-server architecture — three files (index.html, app.js, worker.js) on GitHub Pages; no backend, no auth, no data egress.
Vanilla JavaScript · Transformers.js v2.17.2 · Web Workers API · Tailwind CSS (CDN) · GitHub Pages
Energy Trading · Full CI/CD In Production

EnergyTRM — Energy Trading & Risk Management

Lead Developer

Full-featured ETRM training portal simulating a system like Endur — deal capture, validation, logistics, invoicing, settlement, risk monitoring (VaR, stress tests). Covers physical trades, OTC financial instruments (swaps, futures, options), and counterparty management across 7 complete phases.

energytrm.com ↗ GitHub ↗
  • Two-stage GitHub Actions CI/CD — ci.yml spins up a MySQL 8.0 service container and runs php artisan migrate + php artisan test (PHP 8.4 + Laravel 13) on every PR and develop push. deploy.yml triggers on main, builds frontend assets, and deploys via FTP to cPanel production.
  • Single-table inheritance for financial instruments — financial_trades covers swaps, futures, and options via an instrument_type enum; instrument-specific columns (fixed_rate, strike_price, num_contracts, etc.) are nullable, populated only for the relevant instrument type.
  • Three-ID trade identity pattern — Deal Number (permanent, per instrument), Transaction Number (changes on amendment, shared UNION sequence across trades + financial_trades), and Instrument Number (shared across linked OTC duplicates).
  • State machine per instrument class — physical trades: Pending → Validated → Active → Settled/Closed; futures/options: Pending → Validated → Open → Expired/Exercised/Closed. TERMINAL_STATUSES blocks edits on closed positions.
  • Module-namespaced controller architecture — Master\*, Trades\*, Operations\*, Financials\*, Risk\*, Admin\* namespaces. Static paths (/create, /edit) registered before wildcard {model} routes to prevent shadowing.
  • Environment isolation — .env never committed; production secrets uploaded directly to cPanel. CI uses .env.testing with GitHub Actions service container credentials. cPanel cron runs php artisan schedule:run every minute for daily EoB/CoB checklist resets.
Laravel 13 · PHP 8.4 · MySQL 8.0 · Alpine.js · Bootstrap 5 · Vite · GitHub Actions (CI + FTP deploy) · cPanel
